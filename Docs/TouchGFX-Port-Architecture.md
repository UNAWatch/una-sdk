# TouchGFX Port Architecture

## Overview

The TouchGFX port implementation for the UNA SDK provides a hardware abstraction layer (HAL) that integrates the TouchGFX GUI framework into the UNA platform. This port enables the development of graphical user interfaces for UNA-based applications using TouchGFX's rich widget library and rendering capabilities.

The implementation consists of custom HAL classes that extend TouchGFX-generated code, a command processor for kernel integration, and stub implementations for hardware interfaces not utilized in the current UNA platform configuration. The port supports a 240x240 pixel display with 8-bit color depth using ABGR2222 format, software-based rendering, and button-based input handling through the UNA kernel messaging system.

## Key Classes and Interfaces

### TouchGFXHAL
- **Location**: `Libs/Source/Port/TouchGFX/TouchGFXHAL.cpp`, `Libs/Header/SDK/Port/TouchGFX/TouchGFXHAL.hpp`
- **Purpose**: Main hardware abstraction layer extending TouchGFXGeneratedHAL
- **Key Responsibilities**:
  - Frame buffer management
  - Button controller integration
  - Display initialization and configuration
  - Interrupt handling (delegated to generated HAL)

### TouchGFXCommandProcessor
- **Location**: `Libs/Source/Port/TouchGFX/TouchGFXCommandProcessor.cpp`, `Libs/Header/SDK/Port/TouchGFX/TouchGFXCommandProcessor.hpp`
- **Purpose**: Central command processor and lifecycle manager
- **Key Responsibilities**:
  - Kernel message processing
  - GUI lifecycle management (start/stop/resume/suspend)
  - Button event handling
  - Custom message routing
  - Frame synchronization

### HWButtonController
- **Location**: `Libs/Source/Port/TouchGFX/TouchGFXHAL.cpp`
- **Purpose**: Button input controller for TouchGFX
- **Key Responsibilities**:
  - Sampling button states from command processor
  - Translating UNA button events to TouchGFX key codes

### Generated Classes
- **TouchGFXGeneratedHAL**: Base HAL implementation generated by TouchGFX tools
- **OSWrappers**: Operating system abstraction layer
- **STM32DMA**: DMA interface (stub implementation)
- **STM32TouchController**: Touch input controller (stub implementation)
- **TouchGFXGPIO**: GPIO interface (stub implementation)

## API Structure

### Public Interfaces

#### TouchGFXHAL
```cpp
class TouchGFXHAL : public TouchGFXGeneratedHAL {
public:
    TouchGFXHAL(DMA_Interface& dma, LCD& display, TouchController& tc, uint16_t width, uint16_t height);
    virtual void initialize();
    virtual void disableInterrupts();
    virtual void enableInterrupts();
    virtual void configureInterrupts();
    virtual void enableLCDControllerInterrupt();
    virtual bool beginFrame();
    virtual void endFrame();
    virtual void flushFrameBuffer();
    virtual void flushFrameBuffer(const Rect& rect);
    virtual bool blockCopy(void* dest, const void* src, uint32_t numBytes);

protected:
    virtual uint16_t* getTFTFrameBuffer() const;
    virtual void setTFTFrameBuffer(uint16_t* adr);
};
```

#### TouchGFXCommandProcessor
```cpp
class TouchGFXCommandProcessor {
public:
    static TouchGFXCommandProcessor& GetInstance();
    void setAppLifeCycleCallback(IGuiLifeCycleCallback* cb);
    void setCustomMessageHandler(ICustomMessageHandler* h);

```

### Configuration Functions
- `touchgfx_init()`: Initializes TouchGFX framework
- `touchgfx_taskEntry()`: Main GUI task entry point (never returns)
- `touchgfx_components_init()`: Component initialization (empty)

## Behavioral Architecture

### Initialization Flow
1. **Application Startup**: UNA kernel launches GUI application
2. **TouchGFX Initialization**:
   - `touchgfx_init()` called
   - Bitmap database and text resources registered
   - Font provider configured
   - Frontend heap initialized
   - HAL initialized via `TouchGFXHAL::initialize()`
3. **HAL Setup**:
   - Parent HAL initialization
   - Frame rate compensation enabled
   - Event listener registered
   - Frame buffer allocated (static 240x240 buffer)
   - Button controller initialized

### Rendering Flow
1. **Frame Start**: `HAL::beginFrame()` called
2. **Drawing Operations**: TouchGFX widgets render to frame buffer
3. **Frame Flush**: `TouchGFXHAL::flushFrameBuffer()` called
   - Sets flush request flag
   - Calls generated HAL flush
4. **Display Update**: Command processor sends display update message to kernel
5. **Frame End**: `HAL::endFrame()` called

### Input Handling
1. **Button Events**: Kernel sends `EVENT_BUTTON` messages
2. **Event Processing**: Command processor handles button events
3. **Key Mapping**: Button IDs mapped to TouchGFX key codes:
   - SW1 → '1'
   - SW2 → '3'
   - SW3 → '2'
   - SW4 → '4'
4. **Sampling**: `HWButtonController::sample()` retrieves last button code

### Lifecycle Management
1. **Start**: `onStart()` callback on first frame tick
2. **Resume/Suspend**: Handled via kernel commands
3. **Stop**: `onStop()` callback, application exits
4. **Frame Ticks**: `EVENT_GUI_TICK` messages trigger `onFrame()` callbacks

## Structural Architecture

### Class Hierarchy
```
TouchGFXGeneratedHAL (generated)
    └── TouchGFXHAL (custom)
        ├── HWButtonController (nested class)

TouchGFXCommandProcessor (singleton)
    ├── FixedQueue<MessageBase*, 10> (member)
    └── Kernel (reference)
```

### Key Relationships
- **TouchGFXHAL** extends **TouchGFXGeneratedHAL** for custom behavior
- **TouchGFXCommandProcessor** singleton manages application lifecycle
- **HWButtonController** samples from **TouchGFXCommandProcessor**
- **OSWrappers** delegates VSync waiting to **TouchGFXCommandProcessor**
- Generated classes provide base implementations with minimal functionality

### Data Structures
- **Frame Buffer**: Static `uint8_t sFrameBuffer[57600]` (240×240×1 byte/pixel)
- **Active Buffer Pointer**: `uint8_t* spActiveBuffer`
- **Flush Request Flag**: `bool sFlushBufferReq`
- **Button State**: `uint8_t mLastButtonCode`
- **Message Queue**: `FixedQueue<MessageBase*, 10> mUserQueue`

## Data Flows

### Rendering Data Flow
```
TouchGFX Widgets → Frame Buffer → flushFrameBuffer() → Display Update Message → Kernel → Display Hardware
```

### Input Data Flow
```
Hardware Buttons → Kernel → EVENT_BUTTON Message → TouchGFXCommandProcessor → mLastButtonCode → HWButtonController → TouchGFX
```

### Lifecycle Data Flow
```
Kernel Commands → TouchGFXCommandProcessor → Lifecycle Callbacks → TouchGFX Application
```

### VSync Synchronization
```
Kernel → EVENT_GUI_TICK → TouchGFXCommandProcessor.waitForFrameTick() → OSWrappers.waitForVSync() → TouchGFX Frame Processing
```

## Design Patterns

### Singleton Pattern
- **TouchGFXCommandProcessor**: Single instance manages global GUI state
- Ensures centralized command processing and lifecycle management

### Adapter Pattern
- **TouchGFXHAL**: Adapts TouchGFX HAL interface to UNA platform
- **HWButtonController**: Adapts UNA button events to TouchGFX input

### Template Method Pattern
- **TouchGFXHAL** overrides specific methods while delegating others to generated base class
- Allows customization of key behaviors while maintaining framework compatibility

### Observer Pattern
- **Event Listeners**: HAL registers with TouchGFX Application
- **Lifecycle Callbacks**: Command processor notifies registered callbacks

### Bridge Pattern
- **OSWrappers**: Bridges TouchGFX OS requirements to UNA kernel
- Separates platform-specific OS operations from framework logic

## Overall Architecture Characteristics

### Hardware Configuration
- **Display**: 240×240 pixels, 8-bit color (ABGR2222)
- **Frame Buffer**: Single buffer, software rendering
- **DMA**: Not supported (stub implementation)
- **Touch**: Not implemented (stub controller)
- **GPIO**: Not utilized (stub implementation)

### Performance Characteristics
- **Rendering**: Software-based, no hardware acceleration
- **Memory**: Static frame buffer allocation
- **Synchronization**: Kernel-driven VSync via messaging
- **Input**: Polled button sampling

### Integration Approach
- **Kernel Integration**: Message-based communication
- **Lifecycle Management**: Command processor handles app lifecycle
- **Threading**: Single-threaded GUI execution
- **Resource Management**: Frontend heap for TouchGFX allocations

### Extensibility
- **Custom Messages**: Support for application-specific kernel messages
- **Lifecycle Callbacks**: Configurable start/stop/resume/suspend handlers
- **Hardware Abstraction**: Modular HAL design allows platform customization

### Limitations
- No hardware-accelerated rendering
- No touch input support
- No DMA operations
- Single frame buffer (no double buffering)
- Button-only input (no gesture support)

This architecture provides a solid foundation for TouchGFX integration in the UNA SDK, balancing framework requirements with platform constraints while maintaining clean separation of concerns and extensibility for future enhancements.
